# 작업 흐름 규칙

## 기본 흐름

1. 현재 구현과 관련 지침을 확인합니다.
2. 가장 작은 검증 가능한 성공 기준을 정의합니다.
3. 작업에 필요한 파일만 수정합니다.
4. 가장 구체적이고 유효한 검증을 실행합니다.
5. 결과, 근거, 남은 공백을 보고합니다.

## Swagger/API 작업

API 기반 관리자 기능을 구현하기 전에는 아래 순서를 따릅니다.

1. `.claude/references/api/admin.md`를 읽습니다.
2. 관련 API reference 파일을 읽습니다.
3. 엔드포인트가 없으면 Swagger를 확인하고 reference를 업데이트합니다.
4. API 함수는 `src/api`에 구현합니다.
5. 서버 상태는 TanStack Query hook으로 감쌉니다.
6. 페이지/컴포넌트는 프로젝트 UI 라이브러리 기준으로 구현합니다.
7. 위험도에 따라 집중 테스트, lint, build로 검증합니다.

## 변경 통제

- 관련 없는 정리는 feature diff에 섞지 않습니다.
- production dependency 추가 전에는 확인을 받습니다.
- 기존 사용자 변경을 보존합니다.
- 새 추상화보다 삭제와 재사용을 우선합니다.

## 결정 기록

구조, 라이브러리, 공통 규칙, API 처리 방식처럼 이후 작업에도 영향을 주는 결정을 내렸다면 `.claude/decisions/README.md`를 확인하고 필요한 경우 decision 문서를 추가합니다.

- 기존 규칙을 그대로 따르는 단순 구현은 기록하지 않습니다.
- 결정 문서는 근거와 대안을 짧게 남기는 용도이며 작업 로그로 사용하지 않습니다.
