# 코드 품질 원칙

이 문서는 LLM 코딩에서 흔한 실수를 줄이기 위한 기본 행동 규칙입니다. 빠른 진행보다 정확성, 단순성, 검증 가능성을 우선합니다.

## 1. 코드 작성 전에 생각하기

추측하지 말고, 모호함을 숨기지 말고, 구현 전에 가정과 트레이드오프를 드러냅니다.

- 가정을 명시합니다.
- 해석이 여러 개라면 가능한 해석을 말합니다.
- 더 단순한 접근이 있다면 설명합니다.
- 모호함이 해결 방향을 바꿀 정도라면 수정 전에 질문합니다.

## 2. 단순함 우선

문제를 해결하는 최소한의 코드만 작성합니다.

- 요청하지 않은 기능을 추가하지 않습니다.
- 한 번만 쓰는 코드를 위한 추상화를 만들지 않습니다.
- 요청하지 않은 설정 가능성이나 확장성을 넣지 않습니다.
- 해결책이 문제보다 커졌다면 단순화합니다.

## 3. 필요한 부분만 수정

요청을 해결하는 데 필요한 파일과 줄만 수정합니다.

- 필요한 경우가 아니면 주변 코드를 리팩터링하지 않습니다.
- 기존 스타일, 네이밍, 파일 위치를 따릅니다.
- 이번 변경으로 unused가 된 import, 변수, 함수는 정리합니다.
- 관련 없는 dead code는 삭제하지 말고 언급만 합니다.

모든 변경 라인은 사용자 요청과 직접 연결되어야 합니다.

## 4. 목표 기반 실행

작업을 검증 가능한 목표로 바꿉니다.

```text
1. 가장 작은 동작 변경 구현 -> 대상 검증 실행
2. 관련 테스트 또는 문서 업데이트 -> 영향 경로 확인
3. 최종 검증 실행 -> 근거와 남은 공백 보고
```

`동작하게 만들기`처럼 약한 목표는 명확한 성공 기준으로 바꿔야 합니다.
